# 开发路线与待办清单

本文档用于记录 `webgame_wlsg` 当前阶段的下一步工作和后续迭代方向，避免开发过程中遗漏关键事项。

最后更新：2026-04-02

## 当前项目状态

当前仓库已经完成以下基础整理：

- 默认正式入口已经切到 `/city`
- 侧边栏已拆为模块化组件，主容器不再承载大量混合逻辑
- 地图页历史 warning 已清理
- 基础 Playwright 逻辑冒烟测试已接入
- README 与基础工程说明已更新

当前项目仍然属于“可持续开发的前端原型”，不是功能完整版本。后续工作重点应放在数据流稳定、玩法链路闭环和测试补强，而不是继续无序堆页面。

## 第一优先级：下一步立即处理

这些事项最适合在下一轮开发中优先推进。

### 1. 收口 `gameStore` 职责边界

目标：

- 减少页面组件直接拼装业务数据
- 将“底层状态”和“页面展示态”拆开
- 为后续更多自动化测试和玩法扩展打基础

建议动作：

- 为 `city`、`military`、`map` 提供各自的 view-model / composable
- 将页面里的格式化、组合字段、按钮可用性判断，从组件中迁到 hook 或 selector
- 降低组件直接依赖 `gameStore` 原始字段的范围

### 2. 清理旧调度器残留

目标：

- 明确哪些调度/计时逻辑还有效
- 清掉 Vuex 时代遗留代码
- 避免后续功能接入时误用旧逻辑

建议动作：

- 明确 `useScheduler` 还保留哪些能力
- 把已失效但仍可能被误接入的链路标注或删除
- 将离线收益、升级恢复、征兵恢复、加速恢复的入口统一梳理

### 3. 继续补逻辑测试

当前已有测试覆盖：

- 建筑升级启动与恢复
- 仓库爆仓与容量加成
- 仓库升级恢复
- 征兵入队
- 页面切换后的状态保持

下一批应补：

- 生产力加速的金币扣除与倍率生效
- 征兵完成后的军队入账
- 加速征兵后的剩余时间缩短
- 设置页导出/导入存档
- 地图页关键原型流程的稳定性

## 第二优先级：近期整理

### 4. 拆分军事页数据与交互

当前军事页已经有部分 helper，但还不够彻底。

建议动作：

- 将 `UnitDetailTabs`、`RecruitmentModal`、`RecruitmentQueue` 的共享逻辑继续收进 hook
- 把“兵种展示数据”“征兵成本计算”“队列剩余时间计算”进一步统一
- 为征兵相关逻辑补更明确的测试入口

### 5. 整理地图页的“正式功能”与“测试功能”

当前地图页还混有明显测试性质入口。

建议动作：

- 区分正式地图功能和纯测试入口
- 将测试按钮、模拟器、实验性模块显式标注或迁出正式主界面
- 为 NPC、侦查、攻击等链路定义清晰边界

### 6. 统一存档策略

当前主要依赖 `localStorage`，但不同模块各自写入，后面容易分散。

建议动作：

- 统一所有本地存储 key 的命名和归属
- 明确哪些数据由 `gameStore` 保存，哪些由其他 store 单独保存
- 给存档结构增加版本号，便于未来做兼容迁移

## 第三优先级：中期功能闭环

### 7. 把主循环做完整

目标：

- 让“资源 -> 建筑 -> 征兵 -> 地图行为 -> 战斗结果 -> 资源反馈”形成闭环

建议动作：

- 明确 NPC / 玩家目标、战斗收益、资源掠夺、损耗回写规则
- 统一地图行为与军事系统的状态入口
- 给关键数值配置建立一套可调试、可验证的结构

### 8. 规范通知系统

当前通知已经能用，但还偏分散。

建议动作：

- 按类型统一通知来源
- 避免业务逻辑到处直接拼通知文案
- 给升级完成、征兵完成、资源满仓、加速结束等事件建立统一通知接口

### 9. 处理性能和包体

当前仍有几个明显优化点。

建议动作：

- 继续做页面级懒加载和资源拆分
- 评估是否需要压缩非关键静态资源
- 清理无用依赖和仅开发期依赖的生产构建影响

## 第四优先级：长期方向

### 10. 玩法数值配置化

目标：

- 把主要玩法规则从零散逻辑提炼到清晰配置层
- 便于后续调整平衡性

建议动作：

- 统一建筑、兵种、征兵、战斗、奖励、文明度等配置入口
- 给配置增加校验或最小测试

### 11. 增加更稳定的测试体系

当前只有 e2e 冒烟测试，后面还不够。

建议动作：

- 增加 store 层单元测试
- 增加关键计算函数测试
- 将 Playwright 用例接入发布前检查流程

### 12. 清理开发遗留

建议动作：

- 清理不再使用的演示页、测试页、历史注释和废弃文件
- 保持 README、`docs/`、测试覆盖说明与当前代码一致

## 推荐执行顺序

建议按以下顺序推进：

1. `gameStore` 职责收口
2. 调度器和恢复逻辑清理
3. 补生产力加速与征兵完成测试
4. 军事页进一步模块化
5. 地图页正式/测试功能分层
6. 主玩法闭环
7. 存档版本化与长期测试补强

## 每次开发前建议检查

开始新功能前，先确认以下几点：

- 当前改动是否会碰到 `gameStore` 核心字段
- 是否需要同步补 Playwright 测试
- 是否会影响 `localStorage` 存档结构
- 是否会把测试用模块再次混入正式页面
- 是否需要更新 README 或 `docs/`

## 备注

如果下一轮开发中出现新的优先级变化，应直接更新本文档，不要只停留在对话里。
